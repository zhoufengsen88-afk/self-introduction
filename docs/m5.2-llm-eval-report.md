# M5.2 真实 LLM 回答质量评测

## 1. 目标

M5.2 使用 `m5-agent-qa-v1.jsonl` 调用真实 OpenAI-compatible LLM，形成可复现、可人工复核的回答质量报告。评测重点不是只计算字符串分数，而是区分 Router、检索、生成、拒答和引用支持度各层问题。

## 2. 已实现能力

评测入口：

```bash
make eval-m5-llm
```

底层命令：

```bash
uv run python -m self_intro_api.cli.eval_llm \
  --dataset evals/datasets/m5-agent-qa-v1.jsonl \
  --backend memory \
  --output evals/results/m5.2-llm-memory.json
```

Runner 始终使用 `LLMAnswerGenerator`，不依赖 `ANSWER_GENERATOR` 的取值。模型连接信息从本地 `.env` 读取，结果和控制台均不记录 API Key 或 Authorization Header。

支持的受控运行参数：

```bash
# 只跑一条，适合先验证接口配置
uv run python -m self_intro_api.cli.eval_llm \
  --case-id m5-skillvar-overview-001 \
  --output /tmp/m5-smoke.json

# 只跑前 3 条启用用例
uv run python -m self_intro_api.cli.eval_llm \
  --limit 3 \
  --output /tmp/m5-smoke.json

# 使用已完成入库的 pgvector 链路
uv run python -m self_intro_api.cli.eval_llm \
  --backend pgvector \
  --embedding-provider multilingual-e5-small \
  --output evals/results/m5.2-llm-e5-pgvector.json
```

## 3. 结果结构

每条用例记录：

- 期望和实际 Route；
- 原始回答与结构化拒答状态；
- 返回的完整引用、文档 ID 和 Chunk ID；
- 引用展示、文档命中和 Chunk 命中检查；
- 必须事实的字符串命中信号；
- 禁止事实命中；
- 首 Token 延迟和总延迟；
- Provider 错误码；
- 待人工填写的忠实度、引用支持度和面试表达质量评分。

运行级元数据记录数据集与知识库哈希、模型 ID、温度、检索后端、Embedding、Top-K、Prompt、Router 和 Evidence Policy 版本。结果不记录 Base URL 和 API Key，避免把本地服务信息或密钥写入评测产物。

## 4. 自动指标边界

`required_string_recall` 只用于快速发现明显遗漏，不能代替语义判断。例如模型用“向量与关键词融合”表达“混合检索”时，字符串可能不命中，但语义仍然正确。

`deterministic_checks_pass` 只组合以下可确定检查：

- Route 正确；
- 是否展示引用正确；
- 预期文档和 Chunk 命中；
- 拒答状态与原因正确；
- 未出现明确禁止事实；
- Provider 没有报错。

回答忠实度和“引用是否真正支持关键结论”必须人工复核，不能因为引用 ID 合法就自动判定为正确。

## 5. 人工评分

每条结果预留三个 0～2 分字段：

- `faithfulness_score`：关键事实是否都能由上下文证据支持；
- `citation_support_score`：引用是否真正支持对应结论；
- `interview_quality_score`：回答是否自然、专业、清楚且适合面试场景。

初次运行后，应优先审查自动检查失败、禁止事实命中、拒答错误和低字符串召回用例，再抽查通过样例，防止自动规则漏报。

## 7. 首轮失败分析

### 7.1 Router 错误：4 条

- `m5-ai-assisted-001`：问法缺少当前 Router 的固定经历关键词，被分到 `out_of_scope`。
- `m5-llm-pipeline-001`：问题中的“翻译”先命中通用翻译排除词，覆盖了 Skillvar 知识范围。
- `m5-normal-chat-001`：组合问法“你好，你能回答哪些问题”没有命中精确寒暄集合。
- `m5-normal-chat-002`：“谢谢你的介绍”没有命中精确的“谢谢”。

这说明 Router 不能只靠整句精确匹配；排除词与知识范围同时出现时，也需要明确优先级。

### 7.2 检索与证据错误：5 条

除两条 Router 错误外，以下问题也没有命中正确证据：

- `m5-challenge-001`：没有召回已经明确记录的 GitLab 权限最大难点，模型转而从无关架构 Chunk 推测其他难点。
- `m5-multiturn-001`：多轮中的“最难的一部分”没有稳定映射到 challenge intent。
- `m5-insufficient-001`：AWS 证书问题仍召回 Skillvar 无关 Chunk。

`m5-challenge-001` 是本轮最严重的回答质量失败：知识库已有明确事实，但错误检索让模型声称资料没有标明最大难点，并生成了多个推测候选。这是典型的 retrieval-caused generation failure。

### 7.3 无证据拒答契约错误：1 条

`m5-insufficient-001` 的模型正文正确表示没有 AWS 证书证据，但服务端仍：

- 返回无关 Skillvar 引用；
- `refused=false`；
- 没有 `insufficient_evidence` 原因。

这说明当前“只要检索结果非空就生成”的 Evidence Policy 不足。后续需要相关性门槛或生成后结构化证据不足判定，不能用无关 Top-K 代替有效证据。

### 7.4 生成与展示观察

- 多数命中正确证据的问题能生成自然、专业且职责边界清楚的回答。
- `m5-skillvar-value-001` 在正文中输出了 `SOURCE 1` 等内部上下文标签；最终 UI 已有独立引用卡片，正文不应暴露这些内部标签。
- 部分回答出现证据之外的合理推断，例如把 npm CLI 延伸为 CI/CD 使用场景。这类内容未触发 `forbidden_facts`，但仍需要人工忠实度检查。
- Top-K 8 会让不少回答附带多个未实际支撑正文结论的引用；“引用合法”不等于“引用支持”，后续应收紧证据选择或做 claim-to-citation 对齐。
- 44.93% 的字符串召回与人工阅读差异明显：多条回答语义已覆盖事实，只因措辞不同未命中，不能据此判断模型质量差。

## 8. 下一阶段修复优先级

以下 M5.3 修复已经完成，结果见 `docs/m5.3-query-quality-report.md`：

1. 修 Router：知识范围优先级、组合寒暄、AI 辅助开发表达。
2. 修 challenge 与多轮 query understanding，确保最大难点命中权限文档。
3. 增加检索相关性/Evidence 门槛，让无关 Top-K 触发 `insufficient_evidence`。
4. 强化 Prompt，禁止正文输出内部 `SOURCE N` 标签和证据外推断。
5. 收紧最终引用集合，再进行逐条人工忠实度和引用支持度评分。
