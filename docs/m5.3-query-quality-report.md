# M5.3 查询质量与证据策略优化报告

## 1. 目标

M5.3 根据 M5.2 的真实失败样例修复 Agentic RAG 链路，而不是通过修改问题或更换模型掩盖失败。重点包括：

- Router 对组合寒暄、AI 辅助开发和项目内翻译问题的识别；
- 最大技术难点及多轮指代的检索；
- 无证据个人事实的结构化拒答；
- 上下文和引用范围收敛；
- 职责边界与内部证据标签的生成约束。

## 2. 实现变化

### 2.1 Router

- 支持“你好，你能回答哪些问题”和“谢谢你的介绍”等组合表达。
- 将 Codex、GLM、辅助开发和辅助编程识别为个人经历范围。
- 将项目内部的“LLM 翻译”与“帮我翻译一段文本”区分开。
- 补齐私有代码库、命令行、公开数据以及部分英文知识范围表达。

### 2.2 Query Understanding

新增或增强以下 Retrieval Intent：

- `challenge`：覆盖“最难”“困难”“棘手”和多轮指代。
- `ai_assisted_development`。
- `engineering_delivery`。
- `retrieval` / `retrieval_fallback`。
- `llm_pipeline`。
- `safety_governance`。
- `enterprise_integration`。
- `data_storage`。
- 英文团队/个人职责边界和公开指标表达。

### 2.3 Evidence Policy

- 对“外部认证证书编号”类问题检查查询中的特定英文锚点；知识库没有对应锚点时，直接返回 `insufficient_evidence`，不把无关 Top-K 交给模型。
- 检索仍取 Top-K 8 个候选，但进入上下文和引用的证据最多保留 6 个。
- AWS 证书问题现在返回结构化拒答且没有引用。

该锚点门槛被刻意限制在“证书 + 编号”类型，避免早期过宽规则误拒英文语义问题。

### 2.4 Generation Policy

- Prompt 明确禁止正文输出 `SOURCE N`、Chunk ID 和 Document ID。
- Prompt 明确限制“独立负责全部架构/选型”“生产环境”“彻底解决”等夸大表达。
- 实测发现 AI 辅助开发职责问题仍会突破纯 Prompt 约束，因此该高风险 Intent 改用审核后的 grounded policy answer。
- Router 边界、Evidence 拒答和 AI 职责边界均由确定性 Policy 处理；其余知识问题继续使用真实 LLM。

评测结果逐条记录 `actual_intent` 和 `generation_strategy`。本轮 25 条用例包括：

- 18 条真实 LLM 生成；
- 5 条 Router Policy；
- 1 条 Evidence Policy；
- 1 条 grounded responsibility Policy。

## 5. 关键失败样例修复结果

- AI 辅助开发：由 `out_of_scope` 改为 `knowledge_rag`，最终采用 grounded policy，避免夸大独立决策范围。
- LLM 翻译链路：由 `out_of_scope` 改为 `knowledge_rag`，命中架构与职责证据。
- 最大技术难点：稳定命中 `skillvar-challenges`，回答 GitLab 父级权限继承问题，不再从无关架构推测难点。
- 多轮“那里面最难的一部分”：正确承接 Skillvar 并命中权限难点。
- AWS 证书编号：返回 `insufficient_evidence`、`refused=true`、无引用。
- 组合寒暄与感谢：进入 `normal_chat`，不再错误提示超出范围。
- `SOURCE N`：M5.2 有 1 条正文泄露内部标签，M5.3 为 0 条。

## 6. 项目范围与保留边界

- 本项目是用于面试展示的轻量 Agentic RAG 工程作品，不是正式评测平台，因此不安排 25 条回答的逐条人工打分阶段。
- 结果中的 `human_review_completed: false` 和人工评分字段继续保留，表示没有把自动指标包装成人工质量结论，也方便未来确有需要时扩展。
- 当前已完成重点失败样例抽查，自动回归足以支撑本项目现阶段的开发与演示。
- Citation Hit 只证明预期证据存在于返回集合，不证明每个回答 Claim 都与某个引用完成对齐。
- 证书编号锚点规则是保守的确定性策略，不是通用语义相关性模型。
- 只运行了 memory 检索基线，尚未用 E5 pgvector 重跑 M5.3。
- OpenAI-compatible Provider 当前没有记录 Token Usage 和成本。

## 7. 下一步

不再单独开展人工审核里程碑。下一步优先完成面试演示真正需要的最小可观测性和部署准备，例如请求 Trace、健康检查、Docker、生产环境变量、数据库方案和 SSE 部署验证。P50/P95、多次模型采样、LLM-as-a-Judge 和完整 claim-to-citation 审核不列为当前必做项。
