# M3.5 查询理解优化报告

## 1. 目标

M3.4 语义压力评测已经证明：真实 Embedding 能显著提升英文、同义改写和弱关键词问题的证据召回，但仍暴露出两个问题：

- 只靠向量相似度，部分问题能找到正确文档，却没有稳定命中最关键 Chunk。
- 多轮英文指代、MCP / CLI 分发、检索降级这类问题，需要在检索前补充更明确的意图信息。

因此 M3.5 不继续更换 Embedding 模型，而是先做一层轻量、确定性的查询理解：

```text
原始问题 + 最近历史
→ 意图识别
→ 查询扩展
→ 文档 / 标题 / 内容加权
→ 原有检索与证据组织
```

这一阶段仍不引入 LangChain / LangGraph，也不调用真实 LLM 做 query rewrite。原因是当前问题规模小，确定性规则更容易调试、评测和讲清楚。

## 2. 本次实现

M3.5 新增和调整了 3 类意图：

| 意图 | 解决的问题 | 典型问题 |
| --- | --- | --- |
| `challenge` | 英文多轮指代和权限难点问题无法稳定命中 GitLab 权限 Chunk | `What was the hardest part there?` |
| `mcp_cli` | “自动化工具 / 命令行 / agents”这类弱关键词问题无法稳定命中 MCP 与 CLI Chunk | `这个平台怎么把能力交给自动化工具或命令行使用？` |
| `retrieval_fallback` | 英文检索降级问题能召回正确文档，但回答没有稳定包含降级事实 | `How did the platform behave when the semantic search channel was unavailable?` |

同时修正了意图优先级：

- 先判断权限难点，因为“hardest part / private repository / inherited access”语义很强。
- 职责、团队边界类问题优先于 MCP / CLI，避免把“你负责什么”误判成工具分发问题。
- `retrieval_fallback` 只有同时出现“检索/语义召回相关信号”和“不可用/降级/兜底相关信号”才触发，避免普通“不可用”问题误判。

## 4. 结论

M3.5 说明了一个很重要的工程事实：RAG 质量不只取决于 Embedding 模型。

在当前小知识库里，正确的组合是：

```text
人工审核知识文档
+ 结构化 Chunk
+ E5 语义召回
+ 轻量查询理解 / 意图扩展
+ 可回归评测
```

相比继续盲目换模型，查询理解层更直接地解决了 M3.4 暴露的问题，并且能保留可解释性。

## 5. 边界与下一步

当前 M3.5 仍有明确边界：

- 查询理解是规则驱动，不是真实 LLM query rewrite。
- 评测集只有 13 条语义压力用例，仍需要继续扩充。
- 回答仍由确定性证据组织器生成，还没有接入真实 LLM。
- 当前 rerank 只是文档、标题和关键词加权，还不是 cross-encoder rerank。

下一步建议有两个方向：

1. 进入 M3.6：实现更通用的 rerank / query rewrite，对比规则、LLM 改写和 cross-encoder rerank。
2. 进入 M4：先把当前 RAG 闭环接到 Web MVP，让面试官能在浏览器里真实使用。

我更建议先进入 M4。因为 M3 已经证明端到端 RAG 主链路可用，继续局部优化检索容易变成“实验室打磨”；尽快做出可访问界面，会更符合这个项目作为作品集的目标。
