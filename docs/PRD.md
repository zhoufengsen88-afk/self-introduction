## 2. 产品概述

### 2.1 产品背景

传统简历受到篇幅和表达形式限制，只能高度概括候选人的经历。面试官如果希望进一步了解候选人在具体项目中的职责、技术决策、实施过程、困难、成果和复盘，仍然需要通过有限的面试时间逐项追问。

本项目将候选人的简历、项目文档和个人经历整理成结构化知识库，通过检索增强生成（RAG）、大语言模型和轻量 Agentic 编排，以对话形式向面试官提供基于真实材料的回答。

个人经历只是应用场景，项目本质是一个面向真实知识库的 Agentic RAG 应用。它需要展示候选人对知识工程、Embedding 检索、上下文构造、LLM 生成、Agent 路由、引用校验、评测、可观测性、安全和线上部署的理解与落地能力。

本项目是供面试官访问的个人作品集，不是收费产品或面向企业客户的商业 SaaS。产品优先展示候选人本人、真实项目和工程能力，不以用户增长、付费转化、大规模并发或企业级服务等级为目标。

### 2.2 产品愿景

构建一个可信、可追溯、可持续更新的 Agentic RAG 面试助手，让面试官能够高效了解候选人的真实经验，同时让项目本身体现 Agent 应用开发工程能力。

### 2.3 MVP 产品定义

面试官进入公开网页后，可以输入与候选人相关的问题。系统通过问题理解或 Agent Router 选择合适的检索/回答流程，从个人知识库中检索证据，构造受控上下文，调用 LLM 生成自然回答，并展示引用来源；当资料不足时，系统明确说明无法确认，而不是编造经历。

MVP 的工程目标是一个可稳定公开访问、成本可控并能在面试中讲清楚的 `production-ready Agentic RAG portfolio`，而不是 `enterprise-grade SaaS`。

### 2.4 核心价值

- 对面试官：突破简历篇幅限制，按兴趣探索候选人的真实经历。
- 对候选人：更完整地呈现项目背景、个人职责、技术思考和复盘。
- 对项目展示：形成一个可以公开访问、可以解释设计取舍、可以量化评测的 Agentic RAG 工程案例。

## 3. 目标与非目标

### 3.1 MVP 目标

1. 建立可维护的个人知识库，覆盖简历、技能和至少一个完整项目。
2. 实现从文档导入、解析、切分、向量化、检索到生成回答的完整 RAG 链路。
3. 接入真实 LLM Provider，将检索证据交给模型生成自然回答。
4. 建立轻量 Agentic 编排层，支持问题路由、检索工具调用、拒答策略和后续模拟面试扩展。
5. 为事实性回答提供可查看的引用来源。
6. 在知识库证据不足时拒绝编造个人经历。
7. 支持围绕同一项目进行基础的连续追问。
8. 提供可公开访问的 Web 页面，并完成线上部署。
9. 对检索、生成、耗时、错误和模型成本建立基础可观测能力。
10. 建立一套可重复执行的离线评测集，支持后续迭代对比。
11. 保持低流量个人项目可承担的部署与模型成本。

### 3.2 MVP 非目标

以下能力不纳入首版：

- 多候选人或多租户知识库。
- 注册、登录、付费、订单、订阅和商业运营功能。
- 面向企业客户的 SLA、复杂审计和多地域容灾。
- 多 Agent 协作和高自治开放式工具调用。
- 语音面试和实时数字人。
- 自动联网搜索并补充候选人事实。
- 自动生成或修改正式简历。
- 复杂的后台角色与权限系统。
- 可视化知识管理后台。
- 服务端长期用户画像和跨会话记忆。
- 模型微调。
- 移动端原生应用。
- 微服务、Kubernetes 或面向大规模流量的分布式架构。

非目标不代表永远不做，而是避免在验证核心价值之前扩大范围。

## 4. 目标用户

### 4.1 核心用户：技术面试官

主要诉求：

- 快速了解候选人的技术背景和项目经历。
- 深挖候选人在项目中的实际职责，而不是只看项目概述。
- 了解技术选型、问题定位、方案权衡和最终效果。
- 判断候选人是否具备岗位需要的工程能力。

### 4.2 次要用户

- HR 面试官：了解职业经历、个人优势、求职方向和沟通表达。
- 候选人本人：检查个人材料完整性，并用于模拟面试或项目复盘。
- 潜在合作方：了解候选人的能力范围和过往实践。

MVP 的需求优先级以技术面试官为准。

## 5. 核心使用场景

### 5.1 快速了解候选人

面试官访问页面，通过推荐问题了解候选人的教育、工作、技能和代表项目。

示例问题：

- 请简单介绍一下你自己。
- 你最有代表性的项目是什么？
- 你的主要技术栈是什么？

### 5.2 深挖项目经历

面试官围绕一个项目连续提问，了解背景、职责、架构、技术决策、难点、结果和复盘。

示例问题：

- 你在这个项目中具体负责什么？
- 为什么选择这个技术方案？
- 项目遇到的最大困难是什么？
- 这个结果是如何量化的？

### 5.3 验证能力边界

面试官询问候选人是否具备某项技术或经验。系统必须区分明确证据、有限经验和资料未覆盖三种情况。

示例问题：

- 你是否有高并发系统的实践经验？
- 你在生产环境使用过 Kubernetes 吗？
- 你是否独立负责过完整的 Agent 项目？

### 5.4 处理无法回答的问题

当问题超出知识库范围、涉及敏感内容或无法从材料确认时，系统应说明限制，并在合适时推荐可以继续询问的方向。

---

## 6. 用户主流程

```text
面试官打开网站
  → 查看候选人简介和推荐问题
  → 输入问题或点击推荐问题
  → 系统理解问题并检索个人知识库
  → 系统根据检索证据生成回答
  → 页面流式展示回答
  → 页面展示引用来源
  → 面试官查看来源或继续追问
```

异常流程：

```text
未检索到充分证据
  → 系统明确说明资料中没有足够信息
  → 不补充未经证实的个人事实
  → 推荐相关问题或邀请用户换一种问法
```

## 7. 功能需求

优先级定义：

- P0：MVP 上线必须具备。
- P1：核心闭环稳定后补充。
- P2：后续版本考虑。

### 7.1 访客端

#### FR-01 候选人首页（P0）

页面应展示：

- 候选人姓名或公开称呼。
- 一句话定位。
- 简短个人介绍。
- 主要技能标签。
- 代表项目入口。
- Agentic RAG 助手使用说明和能力边界。

验收标准：

- 用户无需登录即可访问公开页面。
- 用户能够在首页直接进入问答。
- 页面明确说明回答由 AI 基于候选人资料生成。

#### FR-02 推荐问题（P0）

系统应提供一组可点击的推荐问题，帮助首次访问者快速开始。

验收标准：

- 至少提供 6 个推荐问题。
- 推荐问题覆盖个人介绍、项目、职责、技术、挑战和成果。
- 点击问题后自动发起问答。

#### FR-03 自由提问（P0）

用户可以输入自然语言问题并提交。

验收标准：

- 空问题不可提交。
- 提交后展示明确的加载或生成状态。
- 模型或网络异常时展示可理解的错误信息，并允许重试。

#### FR-04 流式回答（P0）

回答应逐步展示，降低用户等待感。

验收标准：

- 后端正常响应时，前端可以持续接收并展示内容。
- 生成完成、失败和中断状态能够被正确区分。

#### FR-05 来源引用（P0）

事实性回答应关联支持该回答的知识库来源。

每条引用至少包含：

- 来源文档或项目名称。
- 章节名称。
- 支持答案的文本片段。
- 内部来源标识。

验收标准：

- 用户可以展开查看引用片段。
- 引用不得指向未参与本次回答的无关材料。
- 不直接公开原始文件的服务器路径、私有地址或敏感信息。

#### FR-06 多轮追问（P0）

系统应在同一会话内理解必要的上下文，例如“这个项目”“当时为什么这么做”等指代表达。

验收标准：

- 在一个会话内，至少支持围绕同一项目进行 3 轮连续追问。
- 新会话不得错误继承旧会话内容。
- 历史消息长度应受控，避免无限增长。
- MVP 会话历史保存在浏览器当前会话中，由前端随请求提交必要历史；后端不建立长期访客档案。

#### FR-07 无依据时拒答（P0）

当知识库没有充分证据时，系统不得虚构候选人的经历、数据、职责或观点。

验收标准：

- 对评测集中明确不存在的经历，回答应表达“资料中没有足够信息”或同等含义。
- 不使用模型通用知识冒充候选人个人经验。
- 可说明相关通用概念，但必须明确其不是候选人经历的证据。

#### FR-08 回答反馈（P2）

用户可以对回答标记“有帮助”或“无帮助”，并可选填原因。

该功能不属于作品集 MVP 上线门槛，只有需要收集真实体验反馈时再实现。

验收标准：

- 反馈与对应回答关联。
- 不要求访客填写个人信息。
- 管理者可以查看汇总结果。

#### FR-09 会话控制（P1）

用户可以新建对话、清空当前会话或停止正在生成的回答。

### 7.2 知识库

#### FR-10 文档导入（P0）

MVP 至少支持 Markdown 文档。PDF 和 DOCX 可在后续迭代加入。

验收标准：

- 通过本地或部署环境中的受控 CLI 命令执行导入，不建设管理后台。
- 导入失败时能够定位到具体文件和失败原因。
- 原始文档与索引数据具有稳定关联。

#### FR-11 文档解析与切分（P0）

系统应优先根据标题和内容结构切分文档，而不是仅按固定字符数截断。

每个知识块至少保存：

- 文档 ID。
- 项目 ID 或知识类别。
- 标题路径和章节。
- 文本内容。
- 来源标识。
- 可见性级别。
- 内容哈希。
- Embedding。
- 创建和更新时间。

#### FR-12 增量更新（P1）

文档发生变化后，系统应根据内容哈希识别新增、修改和删除的知识块，避免每次全量重建。

#### FR-13 检索（P0）

系统应根据用户问题检索最相关的知识块，并将检索结果提供给生成模型。

验收标准：

- 开发或管理环境能够查看 Top-K 检索结果及相关分数。
- 检索支持按项目、知识类别和可见性过滤。
- 未达到最低证据要求时，可以进入拒答流程。

#### FR-14 知识可见性（P0）

知识内容至少支持以下级别：

- `public`：允许用于公开回答并展示引用。
- `private`：不得用于公开问答。

MVP 不使用 `answer_only`，避免内容可以发送给外部模型但不能向访客展示所带来的额外泄露边界。

### 7.3 回答生成

#### FR-15 基于证据生成（P0）

系统提示词应要求模型仅基于本次提供的知识库证据陈述候选人事实。

验收标准：

- 真实 LLM 回答必须由本次检索上下文支持。
- 回答不得引用未提供给模型的知识块。
- 对证据不足、证据冲突或敏感问题应进入拒答策略。

#### FR-16 职责边界（P0）

回答必须区分候选人个人贡献、团队成果和项目整体成果，不得把团队工作全部归因于候选人。

#### FR-17 回答风格（P0）

默认使用第三人称介绍候选人，表达简洁、专业、适合面试场景。用户明确要求第一人称、自我介绍稿或面试口述稿时，可以切换为候选人第一人称。回答深度根据问题调整，不主动堆砌无关技术术语。

#### FR-18 敏感问题处理（P0）

系统不得回答被标记为私密的信息，包括但不限于：

- 身份证件、家庭住址、私人联系方式。
- 账号密钥和访问凭据。
- 前雇主未公开的商业信息。
- 文档中明确标记为不可公开的内容。

#### FR-19 Agentic 问题路由与工具化编排（P0）

系统应根据问题类型选择合适的处理路径，例如个人简介问答、单项目深挖、职责边界、能力边界、无证据拒答或后续模拟面试流程。

验收标准：

- 路由结果可调试、可评测，并能解释为什么选择该路径。
- 首版可以使用确定性 Router 或轻量规则实现，不强制引入 LangChain / LangGraph。
- Router 只能选择项目内定义的安全工具，不能开放任意外部工具调用。
- 权限过滤、Evidence Policy 和引用校验不能交给模型自行决定。

### 7.4 管理与运维

#### FR-20 健康检查（P0）

后端提供健康检查接口，用于判断应用和关键依赖是否可用。

#### FR-21 基础日志（P0）

每次请求应记录：

- Request ID；后续接入 Trace 时可增加 Trace ID。
- 可选的非身份客户端请求标识，不建立用户档案。
- 请求时间和结果状态。
- 使用的模型与配置版本。
- 检索结果标识和分数。
- 检索耗时、首 Token 延迟和总耗时。
- Token 用量或可获得的成本数据。
- 错误类型。

日志不应直接保存密钥和不必要的敏感原文。

#### FR-22 知识库重建（P0）

管理者可以通过明确、可重复执行的命令完成数据库初始化和知识库索引重建。

## 8. 回答原则

所有版本都应遵守以下原则：

1. **真实性**：个人事实必须有知识库证据支持。
2. **可追溯性**：关键事实尽可能提供引用。
3. **职责清晰**：区分个人贡献、团队贡献和项目结果。
4. **不确定性表达**：证据不足时明确说明，不以猜测补全。
5. **隐私优先**：公开回答不得泄露私密或受限制信息。
6. **最小必要回答**：回答当前问题，不暴露无关材料。
7. **对抗防护**：用户指令不能覆盖系统的事实、权限和隐私边界。

## 9. 非功能需求

### 9.1 性能

- 公开页面在正常网络环境下应可快速完成首屏加载。
- 正常模型服务条件下，问答应在数秒内开始流式输出。
- 单次请求应配置总超时，并对外部模型调用配置合理重试。
- 性能目标在首个可运行版本完成基准测试后固化为明确数值。

### 9.2 可用性与可靠性

- 生产环境必须通过 HTTPS 提供服务。
- 应用进程异常退出后应能自动重启。
- 模型、数据库或网络异常不得导致页面永久卡死。
- 数据库结构变更应通过版本化迁移执行。
- 知识源由版本控制保存；向量索引属于可重建数据。生产数据库是否使用托管备份根据平台成本决定。

### 9.3 安全

- 所有模型 API Key 和数据库凭据仅保存在服务端环境变量或密钥系统中。
- 密钥、私有文档和生产配置不得提交到 Git 仓库。
- 公共接口需要基础限流和输入长度限制。
- 对用户输入进行必要的格式校验。
- 知识导入不暴露为公共接口。
- 原始文档默认不允许通过静态路径直接访问。
- 对提示注入和越权索取私密内容建立测试用例。

### 9.4 隐私

- 只收集产品运行必需的数据。
- MVP 默认不在数据库长期保存完整访客对话。
- 浏览器只保留完成连续追问所需的当前会话历史，并提供清空入口。
- 运行日志只记录排障所需的请求元数据，默认不记录完整 Prompt 和回答正文。
- 若后续增加对话或反馈存储，必须先补充隐私说明、保存周期和删除方式。

### 9.5 可维护性

- 本地开发环境可通过清晰文档和少量命令启动。
- 开发、测试和生产配置相互隔离。
- 模型、Embedding 和向量存储通过接口封装，避免业务代码与单一供应商强绑定。
- 核心检索与生成逻辑具备自动化测试。

### 9.6 可观测性

一次错误回答应能够被拆分定位为：

- 文档缺失或内容错误。
- 文档解析或切分错误。
- 检索未命中。
- 排序不合理。
- 上下文构造错误。
- 模型生成不忠实。
- 前端展示或引用关联错误。

## 10. 评测方案

### 10.1 MVP 评测集

首版建立不少于 30 个问题的固定评测集，并逐步扩展到 50 个以上。问题应覆盖：

- 个人基本信息。
- 项目背景和目标。
- 个人职责。
- 技术选型和权衡。
- 难点与解决过程。
- 量化成果。
- 失败和复盘。
- 跨项目比较。
- 知识库中不存在的经历。
- 敏感信息和提示注入。
- 多轮指代和追问。

每条评测数据至少包含：

- 用户问题。
- 问题类型。
- 期望命中的文档或知识块。
- 必须出现的关键事实。
- 禁止出现的错误事实。
- 是否应拒答。
- 备注。

### 10.2 核心指标

| 指标 | 定义 |
| --- | --- |
| 检索命中率 | 期望证据是否出现在 Top-K 检索结果中 |
| 回答忠实度 | 答案中的事实是否得到检索证据支持 |
| 回答相关性 | 回答是否直接解决用户问题 |
| 引用有效率 | 引用内容是否真正支持对应结论 |
| 正确拒答率 | 对无证据或受限制问题是否正确拒答 |
| 首 Token 延迟 | 从提交问题到开始展示回答的时间 |
| 总响应时间 | 从提交问题到生成完成的时间 |
| 单次问答成本 | Embedding、生成和其他模型调用的总成本 |
| 请求错误率 | 请求因应用或依赖异常而失败的比例 |

### 10.3 MVP 初始质量门槛

以下是待基准测试验证的初始目标，不作为未经测量的最终承诺：

- 30 个核心问题全部可以完成请求流程。
- 事实性回答均返回至少一个可核验来源，或明确进入无依据处理。
- 对明确不存在的个人经历，不生成肯定性虚构回答。
- 核心问题的期望证据 Top-K 命中率达到可接受水平，并记录基线。
- 提示注入测试不能获得 `private` 内容或系统密钥。
- 新环境可按照 README 完成本地启动和评测。

## 12. 成功标准

MVP 成功不以“使用了多少 Agent 框架”衡量，而以是否形成可信、可评测、可上线的 Agentic RAG 闭环衡量。

MVP 完成的定义：

- 面试官可以通过公网 HTTPS 地址访问产品。
- 能够围绕至少一个项目完成有引用的问答和连续追问。
- 真实 LLM 能基于检索到的公开 Chunk 生成自然回答。
- Agentic Router 能把常见问题路由到可解释、可评测的处理路径。
- 对知识库外问题能够稳定表达不确定性或拒答。
- 知识文档修改后可以通过可重复流程更新索引。
- 可以查看一次问答的检索证据、耗时和错误信息。
- 固定评测集可以重复运行并保留结果。
- 项目具备部署、配置和故障排查文档。

## 14. 风险与应对

| 风险 | 影响 | 初步应对 |
| --- | --- | --- |
| 知识材料过少 | 回答浅、频繁拒答 | 先完善一个项目的高事实密度资料，再扩大项目数量 |
| 文档内容不准确 | 系统稳定地产生错误回答 | 将知识文档作为唯一事实源并进行人工审核 |
| 切分不合理 | 正确证据无法召回 | 保留标题层级，建立检索评测并观察 Top-K |
| 模型夸大职责 | 损害可信度 | 在资料和提示词中明确个人/团队边界，并加入反例评测 |
| 引用与答案不一致 | 用户无法信任结果 | 保存回答与知识块关联，评测引用支持度 |
| 提示注入 | 泄露私有资料或系统信息 | 可见性过滤在模型调用前执行，增加攻击测试和输出检查 |
| 线上成本失控 | 服务不可持续 | 限流、上下文预算、模型分级、缓存和成本告警 |
| 外部模型不可用 | 问答失败 | 超时、有限重试、友好降级和供应商接口抽象 |
| 过早引入复杂 Agent 框架 | 调试困难、交付延期 | 先实现轻量 Router / Tool / Policy 编排，有明确状态图、暂停恢复或复杂多工具需求后再引入 |
