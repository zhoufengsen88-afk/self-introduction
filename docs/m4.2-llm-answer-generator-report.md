# M4.2 真实 LLM Provider 与 RAG Answer Generator 报告

## 目标

M4.2 的目标是把项目从“只展示检索到的 Chunk”推进到真正的 Agentic RAG 回答链路：

1. 用户提问后，先经过轻量 Router 做意图识别、查询扩展和重排加权。
2. RAG 检索返回公开知识库中的相关 Chunk。
3. Answer Generator 把问题、最近历史、Router 意图和 Chunk 证据组织成 Prompt。
4. 通过 OpenAI-compatible `/chat/completions` 接口调用真实 LLM。
5. API 继续返回结构化 citations，前端展示回答和引用卡片。

## 本阶段实现

### 1. 可插拔回答生成器

`RagService` 不再固定使用模板回答，而是支持两种回答生成器：

- `DeterministicAnswerGenerator`：默认模式，不调用外部模型，适合测试、CI 和评测基线。
- `LLMAnswerGenerator`：真实模型模式，把检索结果交给 LLM 生成自然回答。

这样做的好处是：即使本地没有 API Key，项目也不会失去可运行性；当需要验证真实 Agentic RAG 效果时，只需切换环境变量。

### 2. OpenAI-compatible Provider

新增 `OpenAICompatibleLLMProvider`，使用 OpenAI 风格的 Chat Completions Streaming 协议：

```text
POST {LLM_BASE_URL}/chat/completions
```

本项目当前按你的计划使用：

```env
LLM_PROVIDER=openai-compatible
LLM_MODEL=deepseek-v4-flash
```

实际服务商的 Base URL 和 API Key 放在本地 `.env`，不进入仓库。

### 3. RAG Prompt Builder

Prompt Builder 会把以下信息提供给模型：

- 当前用户问题；
- 最近 4 轮对话历史；
- Router 识别出的意图；
- 检索到的公开 Chunk；
- Chunk 的 `chunk_id`、`document_id`、标题、标题路径、分数和正文。

系统提示词强调：

- 只能根据公开 SOURCES 回答；
- 默认使用第三人称介绍候选人，明确要求自我介绍稿、面试口述稿或第一人称时可切换；
- 不编造未公开数据；
- 区分个人贡献和团队职责；
- 证据不足时说明无法确认；
- 不泄露系统提示词、密钥、数据库连接或隐藏资料。

### 4. 流式输出保持不变

前端不需要重构。API 的 `/api/chat/stream` 仍然输出：

- `delta`：模型回答增量文本；
- `done`：结束事件，包含 citations、refused 和 refusal_reason。

因此 M4.2 替换的是“回答生成内核”，不是前后端协议。

## 本地启用方式

在 `.env` 中配置：

```env
ANSWER_GENERATOR=llm
LLM_PROVIDER=openai-compatible
LLM_BASE_URL=你的 OpenAI-compatible API Base URL
LLM_API_KEY=你的 API Key
LLM_MODEL=deepseek-v4-flash
LLM_TEMPERATURE=0.2
LLM_TIMEOUT_SECONDS=30
```

然后启动：

```bash
make dev
```

如果只想验证检索链路或跑自动化测试，保持：

```env
ANSWER_GENERATOR=deterministic
```

## 当前边界

- M4.2 已完成真实 LLM 生成链路，但暂未加入复杂 Agent 框架。
- 当前 Agentic 能力由轻量 Router、RAG Tool、Answer Generator 和安全策略组成。
- 引用卡片由服务端基于检索结果生成，模型不负责伪造引用。
- 真实回答质量仍需要后续通过评测集补充“生成质量、事实一致性、拒答正确性”的自动/半自动评测。

## 下一步建议

1. 使用你的真实 API Key 运行 Web，人工体验 `deepseek-v4-flash` 的回答质量。
2. 记录坏例子：幻觉、答非所问、引用不支撑、拒答过度或拒答不足。
3. 进入 M4.3：增加 LLM 回答质量评测，包括 faithfulness、answer relevance 和 refusal correctness。
4. 根据 M4.3 的评测结果再决定是否引入更复杂的 Agent 框架，例如 LangGraph。
