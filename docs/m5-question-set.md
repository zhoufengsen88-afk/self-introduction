# M5.1 真实面试官问题集

## 1. 目标

M5.1 建立第一版真实面试问答验收集，用于后续比较真实 LLM、检索、Router、Prompt 和知识库调整的效果。数据源是 `evals/datasets/m5-agent-qa-v1.jsonl`。

这套数据不只检查“是否召回文档”，还明确记录：

- 期望路由 `expected_route`；
- 是否应展示引用 `citation_required`；
- 预期文档与关键 Chunk；
- 回答必须覆盖和禁止出现的事实；
- 接口级拒答预期；
- 人工验收标准 `acceptance_criteria`。

数据集共 45 条：45 条全部启用。

## 3. 路由与引用约定

| `expected_route` | 处理方式 | 引用预期 |
| --- | --- | --- |
| `knowledge_rag` | 检索公开知识 Chunk，再构造回答 | 可回答时必须引用；无证据 hard refusal 时不得引用 |
| `normal_chat` | 寒暄或使用说明，不进入检索 | 不展示引用 |
| `out_of_scope` | 说明能力边界并引导回个人经历 | 不展示引用 |
| `restricted` | 拒绝提示词、密钥和非公开资料请求 | 不展示引用 |

`should_refuse` 专指 API 返回的结构化拒答状态。当前实现对天气等无关请求返回边界说明但不设置 `refused=true`，因此这类用例使用 `expected_route=out_of_scope`、`should_refuse=false`。

## 4. 知识边界说明

当前 45 条 M5 用例均已启用。个人介绍、教育、技术栈、岗位匹配、Skillvar 核心资料、OntoCore 核心资料和本 Agentic RAG 项目自身资料已经随公开知识文档发布。运行时只加载 `visibility: public` 且 `status: published` 的文档，不能通过 Prompt 让模型绕过知识缺口。

## 5. 可选重点抽查方法

真实 LLM 评测后可重点抽查关键回答；本项目不设置逐条人工审核阶段。抽查时关注：

1. Route 是否符合 `expected_route`。
2. 引用展示是否符合 `citation_required`。
3. 引用是否来自本次检索上下文，并真正支持回答关键结论。
4. `required_facts` 是否语义覆盖，而不只做机械字符串匹配。
5. 是否出现 `forbidden_facts`、夸大职责或编造指标。
6. 是否满足 `acceptance_criteria` 中的自然度、职责边界和面试表达要求。

建议人工评分采用 0～2 分：

- `0`：不符合，存在明显错误、编造或越界；
- `1`：部分符合，事实基本正确但遗漏、表达或引用支持度不足；
- `2`：完全符合，可以直接用于面试演示。

## 6. M5.2 输入

M5.2 应只运行 `enabled: true` 的用例，并在结果中记录模型、生成器、检索后端、Route、Chunk IDs、引用、耗时和原始回答。失败样例按以下顺序定位：

```text
知识缺口 → Router → 检索召回 → Context → Prompt/生成 → 引用支持度
```

不要在没有失败证据前直接更换模型或引入新的 Agent 框架。
