# M3.2 pgvector 入库与检索闭环报告

## 1. 目标

M3.2 的目标不是直接追求最佳语义检索效果，而是把 M3 的内存 RAG 闭环推进到更接近线上工程的形态：

- 知识文档和 Chunk 可以正式入库。
- Embedding 有模型名、revision 和维度。
- 数据库查询阶段过滤公开可发布内容。
- pgvector 检索结果可以进入同一套回答、引用和拒答逻辑。
- 可以用评测集记录数据库检索基线。

## 2. 当前实现

| 模块 | 当前实现 |
| --- | --- |
| Embedding Provider | `HashingEmbeddingProvider` |
| Embedding 名称 | `local-hashing-embedding` |
| Embedding revision | `m3.2-v1` |
| 维度 | 384 |
| 数据库 | PostgreSQL + pgvector |
| 入库命令 | `make ingest-db` |
| DB 问答命令 | `make ask-db QUESTION="你在 Skillvar 中具体负责什么？"` |
| DB 评测命令 | `make eval-db` |

## 3. 本地验证结果

临时 PostgreSQL 18 实例验证结果：

```text
documents: 5
chunks: 38
embeddings: 38
public_published_documents: 5
```

重复执行 `ingest-db` 后行数保持不变，说明当前 upsert 路径具备幂等性。

## 5. 边界

- 当前 Hash Embedding 是工程基线，不是最终语义模型。
- 当前回答仍由确定性证据组织器生成，不调用真实 LLM。
- 当前数据量只有 38 个 Chunk，pgvector 使用精确检索即可，不需要 ANN 索引。
- 后续接入真实 Embedding 后，需要重新生成 embedding revision，并重新跑 `eval-db`。
